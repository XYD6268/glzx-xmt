## 项目结构

本项目采用典型的Flask应用结构，主要分为以下几个目录：

- `src/`：存放Python后端代码，核心文件为`app.py`，包含所有路由定义和业务逻辑。
- `templates/`：存放所有HTML模板文件，使用Jinja2模板引擎进行动态渲染。
- `static/`：存放静态资源，包括CSS、JavaScript和图片等。其中`js/`子目录包含前端交互脚本。

前端界面通过Flask的`render_template`函数将模板与数据结合，生成最终的HTML页面返回给用户。

## 添加新的HTML模板并集成到Flask路由

要扩展系统前端界面，首先需要创建新的HTML模板文件，并将其放置在`templates/`目录下。例如，若要添加“作品详情页”或“用户主页”，可分别创建`photo_detail.html`和`user_profile.html`。

接下来，在`src/app.py`中定义对应的Flask路由。以添加“排行榜页面”为例，代码如下：

```python
@app.route('/rankings')
@login_required
def rankings():
    # 获取排行榜数据
    top_photos = Photo.query.filter_by(status=1).order_by(Photo.vote_count.desc()).limit(10).all()
    settings = get_settings()
    current_user = User.query.get(session['user_id']) if 'user_id' in session else None
    return render_template('rankings.html', 
                         photos=top_photos, 
                         settings=settings, 
                         current_user=current_user,
                         show_rankings=settings.show_rankings)
```

此路由使用`@login_required`装饰器确保只有登录用户才能访问，并通过`render_template`函数加载`rankings.html`模板，同时传递必要的数据上下文。

## Jinja2模板继承机制与风格统一

为了确保新页面与现有界面风格统一，应使用Jinja2的模板继承机制。虽然当前项目中未显式定义`base.html`，但可通过分析`index.html`提取公共结构和样式，作为事实上的基础模板。

新创建的模板应复制`index.html`中的`<head>`部分，包括所有CSS样式定义和元标签，以保证视觉一致性。同时，利用Jinja2的块（block）功能实现内容替换。例如，在`rankings.html`中可复用导航结构：

```html
{% extends "base.html" %}

{% block title %}票数排行榜 - {{ contest_title }}{% endblock %}

{% block content %}
<div class="container">
    <div class="title">票数排行榜</div>
    <div class="photo-grid">
        {% for photo in photos %}
        <div class="photo-item">
            <img src="/thumb/{{ photo.id }}" alt="参赛照片">
            <div class="photo-title">{{ photo.title }}</div>
            <div class="photo-info">
                <span>{{ photo.class_name }}</span>
                <span>•</span>
                <span>{{ photo.student_name }}</span>
            </div>
            <div class="photo-vote-row">
                <span>票数：{{ photo.vote_count }}</span>
            </div>
        </div>
        {% endfor %}
    </div>
</div>
{% endblock %}
```

尽管当前模板未使用`{% extends %}`，但开发者应主动重构，提取公共部分到`base.html`，以提高可维护性。

## 在现有页面中插入新功能区块的最佳实践

在现有页面中插入新功能区块（如侧边栏通知）时，应遵循以下最佳实践：

1. **使用条件渲染**：利用Jinja2的`{% if %}`语句，根据用户角色或系统设置决定是否显示特定区块。例如，在`index.html`的用户导航栏中，管理员链接仅对角色大于等于2的用户显示。

2. **保持样式一致性**：新添加的区块应使用与现有页面相同的CSS类名和样式规范。例如，按钮应使用`.nav-btn`或`.photo-vote-btn`类，确保颜色、圆角和阴影效果统一。

3. **响应式设计**：确保新区块在不同屏幕尺寸下表现良好。参考现有CSS媒体查询（`@media`），为移动端优化布局。

4. **最小化侵入性**：优先在页面边缘（如侧边栏、页脚）添加新功能，避免干扰核心内容区域。

例如，要在`index.html`中添加一个通知栏，可在`.container`内顶部添加：

```html
{% if settings.notice_enabled and settings.notice_text %}
<div style="background: #007bff; color: white; padding: 10px; border-radius: 8px; margin-bottom: 20px; text-align: center;">
    {{ settings.notice_text }}
</div>
{% endif %}
```

## 编写或修改JavaScript实现交互逻辑

前端交互逻辑由JavaScript文件实现，如`agreement-modal.js`负责协议弹窗的显示与控制。要实现类似“弹窗协议确认”或“动态加载投票结果”的功能，可参考以下模式：

### 协议弹窗实现分析

`agreement-modal.js`定义了一个`AgreementModal`类，其核心功能包括：

- **异步检查协议需求**：`checkAndShow`方法通过`fetch`请求后端API，判断是否需要显示协议。
- **计时器控制**：`startTimer`方法实现阅读倒计时，确保用户达到最小阅读时间后才能同意。
- **事件绑定**：`bindEvents`方法为“同意”和“不同意”按钮绑定点击事件。
- **状态管理**：通过`isAgreed`标志和`callback`回调函数管理用户选择结果。

### 动态加载投票结果

当前系统已实现动态投票功能。用户点击“投票”按钮时，`votePhoto`函数通过`fetch`向`/vote`接口发送POST请求，成功后更新对应作品的票数显示，无需刷新页面。

### 扩展建议

要添加新交互功能，可创建新的JavaScript文件或扩展现有文件。例如，实现“动态加载更多作品”功能，可在`index.html`底部添加“加载更多”按钮，并绑定事件监听器，通过AJAX请求分页数据并追加到`.photo-grid`容器中。

## 前端安全性与响应式设计适配

### 前端安全性

本系统在前端实施了多项安全措施：

1. **XSS防护**：虽然模板中使用了`{{ }}`进行变量输出，但应确保后端对`Agreement.content`等富文本内容进行适当的转义或过滤，防止恶意脚本注入。

2. **防爬与防作弊**：
   - 禁用右键菜单和开发者工具快捷键（F12, Ctrl+Shift+I等）。
   - 检测开发者工具是否打开，若检测到则弹出警告。
   - 禁用文本选择、拖拽和打印功能，保护内容不被轻易复制。

3. **权限控制**：通过`data-user-role`等隐藏元素传递用户角色，在JavaScript中据此决定功能的可用性。

### 响应式设计适配

系统已实现良好的响应式设计，主要通过CSS媒体查询（`@media`）实现：

- **多断点适配**：定义了`1200px`、`900px`、`600px`和`400px`等多个断点，针对不同设备调整布局。
- **弹性布局**：使用CSS Grid（`.photo-grid`）实现照片网格的自适应排列。
- **字体与间距调整**：在小屏幕上减小字体大小、内边距和外边距，确保内容可读且不拥挤。

开发者在扩展界面时，应遵循现有响应式规则，确保新功能在移动设备上也能正常显示和操作。
